логики и клиентских вызовов для различных типов RPC


## 📌 Задание: 
Реализовать gRPC-сервис MusicPlayer с методом: StreamPlaylist(PlaylistID)
Метод должен возвращать поток треков из плейлиста (Server streaming RPC).

---

### Компоненты системы

Client (client.py)  
Отправляет запросы серверу и получает поток треков.

Server (server.py)  
Обрабатывает запросы клиента и отправляет поток данных.

music.proto  
Контракт взаимодействия между клиентом и сервером. Описывает сервис и структуры сообщений.

playlist_db  
Имитация базы данных (список треков в оперативной памяти).

---

## 🧩 Описание реализации

В рамках лабораторной работы был создан gRPC-сервис MusicPlayer.

Описание сервиса выполнено в файле:

music.proto

Сервис содержит метод:

- StreamPlaylist — возвращает поток треков по ID плейлиста.

Метод использует ключевое слово stream, что позволяет серверу отправлять несколько сообщений в ответ на один запрос клиента.

⚙️ Генерация gRPC-кода

После создания music.proto была выполнена команда:

``` python -m grpc_tools.protoc -I. --python_out=. --grpc_python_out=. music.proto
```

В результате были автоматически созданы файлы:
 • music_pb2.py
 • music_pb2_grpc.py

Они обеспечивают взаимодействие клиента и сервера через gRPC.


🖥 Реализация сервера

Серверная часть реализована в файле:

server.py

Создан класс MusicPlayerServicer, наследующийся от MusicPlayerServicer.

Реализован метод:
 • StreamPlaylist
 • принимает ID плейлиста
 • возвращает поток треков с использованием yield
 • используется задержка для имитации стриминга

Сервер запускается на порту:
50051
и ожидает подключения клиентов.

💻 Реализация клиента

Клиентская часть реализована в файле:

client.py

Подключение выполняется через:

grpc.insecure_channel('localhost:50051')

Клиент позволяет:
 • получить поток треков из плейлиста

Получение данных реализовано через цикл for, который обрабатывает поток ответов сервера.
