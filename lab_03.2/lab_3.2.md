# 🔬 Методология

В работе использовалась симуляция Gossip-протокола для распределенной системы, состоящей из 100 узлов.

В процессе моделирования:

* создавалась сеть из 100 узлов;
* моделировался массовый отказ 30% узлов;
* через 60 секунд выполнялось восстановление отказавших узлов;
* измерялось время повторной конвергенции сети;
* определялась нагрузка на сеть;
* оценивалась доля успешно восстановленных узлов.

Исследовалось влияние параметра **Gossip Fanout**, определяющего количество соседних узлов, которым передается gossip-сообщение.

Для сравнения использовались два значения:

* Fanout = 3;
* Fanout = 5.

### Время повторной конвергенции после восстановления узлов

### Анализ графика

На графике показано время, необходимое Gossip-протоколу для повторной синхронизации системы после восстановления отказавших узлов.
<img width="552" height="431" alt="image" src="https://github.com/user-attachments/assets/6a4b4772-b372-446b-82d4-111f47c5401d" />

### Наблюдения

* При **Fanout = 3** время конвергенции составило **0,6 с**.
* При **Fanout = 5** время конвергенции снизилось до **0,2 с**.
* Увеличение Fanout позволило значительно ускорить восстановление системы.

### Причина

При большем значении Fanout каждый узел распространяет информацию большему числу соседей за один цикл Gossip-обмена.

Это позволяет быстрее передать информацию о восстановлении узлов по всей сети и уменьшает время достижения согласованного состояния.

### Вывод

Увеличение Fanout с 3 до 5 позволило сократить время повторной конвергенции примерно на 67%.

# 📝 Выводы по работе

В ходе лабораторной работы была исследована работа Gossip-протокола в условиях массового отказа узлов распределенной системы.

В процессе моделирования анализировалось влияние параметра Gossip Fanout на скорость восстановления системы после отказа 30% узлов и последующего возврата их в сеть через 60 секунд.

Результаты симуляции показали, что Gossip-протокол успешно восстанавливает работоспособность сети даже после значительных отказов. Во всех экспериментах было достигнуто 100%-ное восстановление узлов.

Увеличение значения Gossip Fanout с 3 до 5 позволило сократить время повторной конвергенции с 0.6 до 0.2 секунды, уменьшить количество сообщений с 567 до 435 и снизить пропускную способность сети с 0.56 до 0.43 Мбит/с.

Наиболее эффективной оказалась конфигурация:

* Gossip Interval — 0.2 с;
* Gossip Fanout — 5;
* Packet Loss — 5%;
* Node Failures — 30%.

Полученные результаты подтверждают высокую отказоустойчивость Gossip-протокола и его способность автоматически восстанавливать согласованное состояние распределенной системы после массовых отказов без использования централизованных механизмов управления.
